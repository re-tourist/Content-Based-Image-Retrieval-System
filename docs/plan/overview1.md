# 图像检索系统开发总纲

## 1. 文档目的

本文档用于作为本学期视觉实践课“图像检索系统”项目的开发总参考，目标不是给出一份僵硬的最终方案，而是提供：

1. 一个稳定的工程思考顺序；
2. 一套能随开发逐步细化的系统框架；
3. 一组在开发前就应记录的风险点、技术分层和实验方向；
4. 一条从“现在立刻开工”到“期中验收/期末验收”的落地路线。

本文档基于以下共识：

- 项目目标不是只做一种特殊情形，而是做一个**混合型图像检索系统**；
- 系统应同时考虑**全局语义相似**与**局部几何一致性**；
- 目标是做出一个**工程上完整、可演示、可扩展、可写报告**的系统；
- 项目开发应优先追求**可跑通的最小闭环**，再逐步增强，而不是一开始堆满全部高级模块。

------

## 2. 项目目标重新定义

### 2.1 为什么不能只做单一路线

单一路线往往只能在某一种情形下表现很好：

- 仅做全局深度特征：对语义相似图像较强，但对近重复、局部裁剪、遮挡、视角变化可能不够稳；
- 仅做局部特征 + 倒排：对近重复很强，但对语义检索、跨外观变化、整体语义相关性较弱；
- 仅做某个炫酷模型：容易在答辩时显得“像调库”，而不是一个完整系统。

因此，项目更合理的目标是：

> 构建一个“多路召回 + 强重排 + 可扩展反馈”的混合型图像检索系统。

### 2.2 本项目建议的核心目标

系统应具备以下能力：

1. 能处理一般语义相似图像检索；
2. 能处理近重复 / 局部裁剪 / 遮挡等更严格的实例相似；
3. 能在结果展示中给出一定可解释性；
4. 能逐步扩展到 re-rank、query expansion、深度学习特征等高级模块；
5. 能与老师PPT中的阶段工期对应，便于课程节奏管理。

------

## 3. 从老师PPT抽取出的开发主线

根据PPT，课程时间线大致为：

- **1-2：系统搭建、兴趣点检测和局部特征描述**
- **3-4：特征编码**
- **5-7：倒排索引**
- **8：期中系统验收**
- **9-10：重排序和查询扩展**
- **11-12：哈希**
- **13-15：基于深度学习的图像检索**
- **16：期末系统验收**

PPT给出的传统框架是：

输入图片 → 兴趣点检测和局部特征描述 → visual word / tf-idf表示 → 在倒排表中查询 → Re-rank → Query expansion

这个框架本质上是一个经典的 near-duplicate retrieval pipeline。其特点是：

- 局部特征负责提供局部匹配能力；
- visual word / tf-idf 负责把局部特征转为可检索表示；
- 倒排索引负责高效候选召回；
- re-rank 负责提高前排精度；
- query expansion 负责在已有高置信结果基础上进一步扩大召回。

### 3.1 对PPT框架的现代化理解

如果只按传统路线做，系统会比较经典，但不一定“混合极致”。

因此，更好的理解方式是：

- **传统主线**保留：局部特征、编码、倒排、重排、QE；
- **现代增强主线**并行加入：全局深度特征、ANN检索、后期融合。

也就是说，本项目建议不是推翻PPT，而是：

> 以PPT传统框架为课程主线，以深度全局特征为现代增强支线，最终形成混合系统。

## 4. 系统总架构建议

## 4.1 总体分层

建议将系统拆为三层：

### 第1层：Recall（召回层）

目标：尽量不漏掉真正相关的候选。

候选来源可以有两路：

1. **局部特征 → visual words → tf-idf → 倒排索引召回**
2. **全局深度特征 → 向量索引（ANN/暴力检索）召回**

### 第2层：Re-rank（重排层）

目标：对召回得到的候选进行更精细判断，提高前几名结果的准确性。

可用证据包括：

- 全局向量相似度；
- 局部匹配数；
- RANSAC几何验证内点数/内点比例；
- 稀疏检索分数与稠密检索分数的融合。

### 第3层：Feedback / Expansion（反馈增强层）

目标：利用高置信结果进一步改写查询，提高整体召回或排序质量。

可包含：

- Query Expansion；
- 伪相关反馈（Pseudo Relevance Feedback）；
- 后期哈希或特征压缩增强。

## 5. 为什么要做混合系统

### 5.1 全局特征的优缺点

全局深度特征擅长回答：

> 这张图整体上像不像另一张图？

优点：

- 语义能力强；
- 对颜色、光照、小变形鲁棒；
- 工程实现较直接；
- 对现代演示更友好。

缺点：

- 容易把“长得像但不是同一实例”的图排得很靠前；
- 对局部裁剪、局部遮挡、近重复识别不一定足够稳；
- 可解释性相对弱。

### 5.2 局部特征 + 倒排的优缺点

局部特征路线擅长回答：

> 它们是否真的共享局部结构，能不能对上？

优点：

- 对近重复、裁剪、局部遮挡、视角变化更敏感；
- 易于给出匹配点等可解释结果；
- 与PPT主线完全一致。

缺点：

- 单靠这一路线，对一般语义相似图像不够强；
- 工程复杂度更高；
- 特征编码、词袋、倒排、重排链条较长。

### 5.3 混合的根本理由

混合不是因为“模块越多越强”，而是因为两类证据互补：

- **全局特征**更像“宽召回证据”；
- **局部几何证据**更像“严裁决证据”。

系统最终应做到：

> 用全局特征保证广泛召回，用局部匹配和几何验证保证前排结果可信。

## 6. 项目开发顺序：正确的工程思维

本项目不应一开始就直接写“最终版本”。

正确的顺序应该是：

### Phase A：先建立最小闭环

最小闭环定义为：

> 数据读取 → 特征提取 → 建立索引/表示 → 输入query → 返回TopK结果

注意：

- 这一步不要求很强；
- 但必须真实跑通端到端；
- 没有最小闭环，后续所有优化都没有落点。

### Phase B：先把传统主线打通

由于老师PPT明确给出了课程路线，因此推荐先按传统主线实现：

1. 兴趣点检测 / 局部特征描述；
2. 特征编码（visual words / codebook）；
3. tf-idf表示；
4. 倒排索引查询；
5. 得到可工作的传统检索系统。

### Phase C：再引入现代增强

在传统主线可工作后，引入：

1. 全局深度特征提取；
2. 稠密向量检索；
3. 与传统召回融合；
4. 重排与查询扩展增强。

### Phase D：最后做性能优化与展示优化

包括：

- 哈希 / 压缩 / 加速；
- UI展示；
- 匹配可视化；
- 实验记录与消融分析。

## 7. 开发前必须写进文档或Issue的风险点

以下问题现在不必马上解决，但必须记录，因为它们会在后面真实出现。

### 7.1 任务定义风险

- 系统最终偏向近重复、实例检索还是语义检索？
- 课程评分是看演示效果、检索精度，还是算法完整度？
- 数据集规模是否固定？是否允许自选数据？

### 7.2 表征风险

- 全局特征会不会把“语义相似但不同实例”的图排太前？
- 局部特征在低纹理图像上是否不稳定？
- 重复纹理（窗户、砖墙）是否导致误匹配？

### 7.3 检索风险

- 倒排召回是否对词袋规模敏感？
- 稠密向量检索是否会受距离度量或归一化影响？
- 融合时不同得分量纲不同，如何归一化？

### 7.4 重排风险

- RANSAC 几何验证是否对参数敏感？
- 候选数量过大是否导致重排太慢？
- 若query是大面积裁剪，重排是否仍稳？

### 7.5 扩展风险

- Query Expansion 可能引入漂移，把查询带偏；
- 若错误结果被高置信纳入扩展，会形成误差放大。

### 7.6 工程风险

- 项目可能过早进入“堆模块”状态，缺少统一接口；
- 缺少实验记录导致后续无法复现；
- 缺少评价协议导致“感觉变强了”但实际上无证据。

------

### 8.1 模块接口思维

每个模块都尽量有明确输入输出，例如：

- `extract_local_features`：输入图片路径，输出 keypoints + descriptors；
- `train_codebook`：输入局部描述子集合，输出 visual vocabulary；
- `build_bow_index`：输入 visual words / tf-idf，输出倒排索引；
- `extract_global_features`：输入图片，输出全局向量；
- `build_dense_index`：输入全局向量，输出ANN/矩阵索引；
- `rerank`：输入候选列表 + query特征，输出重排后的列表。

只要接口清楚，后期替换算法就容易。

## 9. 阶段性技术路线建议（与课程工期对齐）

## 第1阶段（1-2周）：系统搭建 + 局部特征

### 目标

### 可交付物

### 适合先用的技术

- OpenCV先快速验证流程。

> 注：第一阶段不追求“最终最强特征”，而追求**系统骨架和最小闭环**。

------

## 第2阶段（3-4周）：特征编码

### 目标

- 把局部描述子转为可检索表示；
- 建立 visual vocabulary；
- 实现 BoW / tf-idf 表示。

### 需要理解的核心问题

- 为什么局部特征不能直接全库暴力匹配？
- 为什么要把连续描述子离散化为 visual words？
- tf-idf 的本质是如何给“有区分度的词”更高权重？

### 可交付物

- codebook训练脚本；
- 图像的visual words表示；
- tf-idf向量保存结果。

------

## 第3阶段（5-7周）：倒排索引

### 目标

- 基于visual words / tf-idf建立倒排表；
- 支持输入query后高效召回候选；
- 能展示初始TopK结果。

### 可交付物

- 倒排表构建程序；
- 检索demo；
- 初始召回结果展示；
- 期中前的系统可运行版本。

### 到期中验收前至少应达到

> 有一个真实可运行的传统检索系统：
> 局部特征 → 编码 → tf-idf → 倒排召回 → 输出检索结果。

------

## 第4阶段（9-10周）：重排序 + 查询扩展

### 目标

- 在倒排召回候选上做更精细判断；
- 引入局部匹配与几何一致性；
- 尝试Query Expansion。

### 建议重点

- 重排优先级高于QE；
- 优先做匹配点 + RANSAC内点可视化；
- QE应当保守，只对高置信结果使用。

### 可交付物

- 重排序模块；
- 几何验证结果；
- QE前后对比实验。

------

## 第5阶段（11-12周）：哈希/加速

### 目标

- 关注检索效率和存储效率；
- 为后续深度特征或混合系统做性能准备。

### 可交付物

- 一版特征压缩/哈希实验；
- 对比不同检索速度与内存消耗。

------

## 第6阶段（13-15周）：深度学习图像检索

### 目标

- 引入全局深度特征；
- 建立第二路检索管线；
- 与传统检索系统融合，形成混合系统。

### 这一阶段建议思路

不是简单“替换掉传统方法”，而是：

> 保留传统局部路线，新增全局深度路线，然后做融合。

### 可交付物

- 全局特征提取模块；
- 深度检索结果；
- 混合检索实验；
- 融合前后对比。

------

## 第7阶段（期末前）：系统收束

### 目标

- 把系统做成完整可演示版本；
- 整理文档、实验、失败案例和改进分析；
- 形成“工程闭环 + 算法闭环 + 报告闭环”。

### 最终系统理想形态

- 传统召回可跑；
- 深度召回可跑；
- 至少一种重排策略可跑；
- 有一定可解释性；
- 有实验对比与阶段性演化脉络。
