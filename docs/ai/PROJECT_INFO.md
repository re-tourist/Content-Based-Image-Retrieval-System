# PROJECT_INFO.md

## 1. 项目基础信息

### 1.1 项目名称

Hybrid Image Retrieval System（混合型图像检索系统）

### 1.2 项目类型

本项目是一个面向课程实践的图像检索系统工程项目。

它不是单一算法复现任务，也不是纯论文阅读任务，而是一个需要同时兼顾：

- 课程工期要求；
- 工程架构设计；
- 传统图像检索主线实现；
- 深度学习增强路线；
- 后续文档、实验、答辩展示；
- GitHub 工程管理与 issue / milestone 工作流；
- 可运行、可扩展、可解释的系统交付。

### 1.3 项目目标（一句话版）

构建一个以传统图像检索主线为基础，并逐步引入深度全局特征与混合召回策略的混合型图像检索系统。

### 1.4 项目目标（详细版）

本项目的目标不是只完成某一个检索算法模块，而是构建一个较完整的课程项目系统，要求其具备：

1. **传统图像检索能力**：
   - 兴趣点检测
   - 局部特征描述
   - 特征编码
   - TF-IDF 表示
   - 倒排索引检索
   - 重排序
   - 查询扩展

2. **现代增强能力**：
   - 深度全局特征提取
   - 稠密向量检索
   - 混合召回与融合

3. **工程能力**：
   - 清晰的模块边界
   - 稳定的目录结构
   - 配置系统
   - 实验记录
   - 可维护的 issue / branch / commit 工作流

4. **展示与答辩能力**：
   - 可运行 demo
   - 关键点可视化
   - 检索结果展示
   - 实验对比
   - 失败案例与改进分析

## 2. 项目问题定义

### 2.1 任务本质

图像检索的核心问题是：

> 给定一张 query 图像，从 gallery 图像库中检索出与其最相关的图像，并按相关性排序返回。

### 2.2 本项目不采用的狭义理解

本项目不希望只做某一种过于特殊的检索情形，例如：

- 只做 near-duplicate（近重复）检索；
- 只做语义检索；
- 只做局部特征传统方法；
- 只做深度 embedding 检索。

### 2.3 本项目采用的理解

本项目采用“混合型图像检索系统”的理解：

- 传统局部特征路线负责实例级/结构级证据；
- 深度全局特征路线负责整体语义/外观证据；
- 后期通过候选融合、重排和增强模块实现更强的综合检索效果。

### 2.4 为什么选择混合系统

单一路线各有明显局限：

- **仅局部特征 + 倒排索引**：
  对近重复、裁剪、遮挡有优势，但对高层语义表达较弱；

- **仅深度全局特征**：
  对整体语义和视觉外观较强，但对细粒度实例对应、几何一致性和局部严格匹配不够稳；

因此，本项目最终目标是：

> 以传统主线为课程实现基础，以深度全局特征为增强路线，最终形成混合型图像检索系统。

## 3. 当前对项目的总体理解

### 3.1 项目定位

这是一个**工程型课程项目**，不是单纯的算法作业。

因此在帮助本项目时，应始终同时考虑：

- 算法正确性；
- 工程可落地性；
- 阶段性工期；
- 模块依赖关系；
- 文档与 GitHub 工作流；
- 最终展示与答辩。

### 3.2 当前开发哲学

本项目已经明确采用以下开发哲学：

#### 上层：宏观规划驱动

先写清：

- 项目目标
- 总体 pipeline
- 宏观工期规划
- 模块划分
- issue 拆分策略

#### 下层：阶段闭环驱动

每个工期只做当前阶段最小可运行闭环，不提前开发后续复杂模块。

### 3.3 当前的关键原则

帮助本项目时应遵守以下原则：

1. **优先考虑课程工期主线**；
2. **优先保证当前阶段的最小闭环可运行**；
3. **先有稳定骨架，再做增强模块**；
4. **先有传统检索可运行版本，再做深度增强与混合系统**；
5. **避免过早设计最终复杂系统**；
6. **文档、issue、branch、commit 应与开发同步推进**。

## 4. 老师给出的课程工期主线

根据老师 PPT，目前课程主线大致为：

- 1-2：系统搭建、兴趣点检测和局部特征描述
- 3-4：特征编码
- 5-7：倒排索引
- 8：期中系统验收
- 9-10：重排序和查询扩展
- 11-12：哈希
- 13-15：基于深度学习的图像检索
- 16：期末系统验收

### 4.1 从课程主线抽象出的系统 pipeline

老师 PPT 中的传统框架大致为：

输入图片 → 兴趣点检测与局部特征描述 → visual words / TF-IDF 表示 → 倒排表查询 → Re-rank → Query Expansion

### 4.2 对此框架的现代化理解

本项目不应抛弃课程主线，而应：

- 保留这条传统检索主线；
- 后期再叠加深度全局特征检索路线；
- 最终形成“传统主线 + 深度增强支线”的混合系统。

## 9. 当前项目最重要的技术路线理解

### 9.1 为什么不是先直接做最终混合系统

因为当前项目仍处于早期阶段，工程骨架和传统主线尚未跑通。

如果一开始就直接做：

- 全局深度特征
- 稠密召回
- 混合融合
- 重排序
- 查询扩展

会导致工程复杂度过高，且与课程工期主线不一致。

### 9.2 当前正确开发顺序

应采用如下顺序：

1. 工程骨架搭建
2. 配置系统
3. 数据层整理
4. 局部特征提取
5. 特征编码
6. TF-IDF 与倒排索引
7. 传统检索闭环
8. 重排序与查询扩展
9. 深度全局特征路线
10. 混合召回与融合

### 9.3 当前最重要的工程判断

帮助本项目时，若面对“现在做什么”这个问题，应优先判断：

> 当前工期最小闭环是什么？

而不是优先讨论最终最复杂版本。

## 10. 当前最小闭环定义

### 10.1 当前工期0-3的最小闭环

当前阶段最重要的闭环不是“完整检索系统”，而是：

> 输入一批图像 → 成功完成数据读取和基础预处理 → 提取局部特征 → 保存特征 → 输出关键点可视化

### 10.2 为什么这是当前闭环

因为它：

- 对齐老师 PPT 的前期工期；
- 是后续特征编码和倒排索引的共同前提；
- 能迫使工程接口、路径、输出格式先稳定下来；
- 难度适中，适合作为真正的项目起点。

## 11. 当前已知风险与开发注意事项

帮助本项目时，请始终考虑下列风险，而不是只给理想化方案。

### 11.1 工程风险

- 过早堆砌复杂模块；
- 没有统一接口导致后续模块难接；
- 文档不同步，导致路线混乱；
- 代码能跑但难以复现。

### 11.2 配置系统风险

- 初期把配置系统设计得过重；
- 过早把后期所有参数塞进 config；
- 引入过度复杂的配置框架。

当前对配置系统的理解是：

> 先建立“最小可用、易扩展”的配置系统，而不是设计最终版配置中心。

### 11.3 技术路线风险

- 过早偏离老师课程主线；
- 先做深度路线而传统主线未跑通；
- 过早讨论最终融合细节而当前基础模块未落地。

### 11.4 评估风险

- 只看 demo，不看可复现的中间结果；
- 只谈系统构想，不明确阶段交付物；
- 没有及时记录失败案例和实验现象。

1. 优先判断问题属于哪个工期；

### 12.3 文档输出偏好

当需要写文档时，希望 AI 产出：

- 可直接放入仓库的 Markdown 内容；
- 有层次、有标题；
- 术语统一；
- 既能作为内部开发文档，也能作为后续答辩素材基础；
- 避免纯口语化碎片输出。

### 12.4 对 Codex / AI coder 提示词的偏好

当生成给 Codex 或其他 AI coder 的提示词时，希望：

- 任务边界明确；
- 强调最小可用实现；
- 不要过度设计；
- 给出具体文件路径；
- 给出实现原则、验收标准和禁止事项；
