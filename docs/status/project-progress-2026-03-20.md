当前项目**已经完成工期0的大部分目标**，包括仓库骨架、核心规划文档、基础配置文件、输出目录约定和模块目录预留。

综合判断：

> **主工程实际进度位于“工期0已基本完成，工期1部分完成但未验收通过，工期3存在独立 POC 验证”的状态。**

如果必须给出一个最接近的工期位置，我认为项目目前应归类为：

> **处于工期1后半段，尚未正式进入工期2。**

## 2. 分析方法

本报告按以下思路判断进度：

1. 读取 `docs/plan/overview.md`，提取工期0-13的目标、主要内容和验收标准。
2. 审查仓库目录、源码、配置、数据和输出目录，确认哪些内容只是规划，哪些内容已经落地。

4. 以“是否满足该工期验收标准”作为最终判定依据，而不是只看是否有零散代码。

## 5. 当前项目到底开发到哪

如果只从“写了什么文档和代码”来看，容易误以为项目已经到了工期2-3。

但如果严格按 `overview.md` 的**验收标准**来看，当前最稳妥的结论是：

1. 工期0基本完成。
2. 工期1部分完成，但还不具备验收条件。
3. 工期2未开始。
4. 工期3只有独立原型验证，不算主工程完成。

因此，当前项目的真实位置应表述为：

> **主线进度仍停留在工期1，且尚未完成；技术探索层面已经摸到工期3。**

## 6. 与计划文档的偏差

当前仓库最明显的偏差不在“方向错误”，而在“落地断层”：

- 文档规划非常完整；
- 目录骨架已经搭好；
- 局部特征的作业型原型已经跑过；
- 但主工程中间层没有接上。

这种偏差会导致一个典型问题：

> **你已经知道后面要做什么，也验证过局部特征能跑，但主工程主线还没有形成真正可迭代的第一段闭环。**

也就是说，当前最缺的不是新想法，而是把已有认知与原型能力迁移进主工程。

## 7. 风险判断

当前阶段最大的风险有四个：

1. **误判进度风险**
   - 容易把 `coursework/week01` 的完成度误当成主工程已经进入工期3。
2. **文档领先代码风险**
   - 规划远超实现，后续若不尽快补齐主线，项目会长期停留在“知道怎么做，但仓库还没做”。
3. **原型与主线分叉风险**
   - 若后续继续只在 `coursework/` 下试验，主工程会越来越空，最终难以答辩时展示统一系统。
4. **缺少状态文档风险**
   - 之前没有一个专门记录“当前做到哪”的固定文档，停一段时间后就很容易失去上下文。

## 8. 建议的下一阶段工作

当前最合理的推进策略不是继续扩展新工期，而是**先补齐工期1并把工期3原型迁移进主工程**。

建议按下面顺序推进：

1. 完成工期1收尾
   - 在 `scripts/` 下增加主工程入口脚本
   - 在 `src/datasets/` 中实现最小数据读取
   - 在 `src/features/local/` 中建立主工程版局部特征接口
2. 启动工期2
   - 统一 query / gallery / train / test 组织方式
   - 生成 split 文件和数据清单
   - 实现基础格式检查与预处理脚本
3. 把原型迁移为主线能力
   - 把 `coursework/week01/core.py` 中可复用的局部特征逻辑抽入 `src/`
   - 让输出进入 `outputs/features/` 与 `outputs/figures/`
4. 再正式进入工期3验收
   - 批量提取特征
   - 保存标准化 descriptors
   - 输出关键点可视化
